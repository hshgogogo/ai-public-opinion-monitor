# 测试、迭代与治理

## Skill 不是写完即完成

Skill 是活文档。每一次误触发、漏触发、输出偏差、权限过宽或用户反复纠正，都是回写和迭代信号。

迭代循环：

```text
发现问题 -> 定位原因 -> 修改 Skill -> 验证效果 -> 记录经验
```

常见定位：

- 误触发：`description` 过宽或缺少“不适用”。
- 漏触发：触发词不足、目录/文件名错误、预算超限。
- 输出不稳：模板缺失、流程步骤模糊、示例不足。
- 计算错误：应脚本化而没有脚本化。
- 行动越界：`allowed-tools` 过宽或任务型未禁自动触发。

## 触发测试

为每个 Skill 准备 10 个应触发和 10 个不应触发用例。初期可以先用 3+3 小集快速验证。

目标：

- 相关任务触发率高于 90%。
- 无关任务误触发率低于 5%。

测试表：

| 用户说法 | 期望 | 实际 | 结论 | 修复动作 |

## 功能测试

验证 Skill 加载后是否真正提升质量：

- 是否按流程执行？
- 是否读取了正确的参考文件？
- 是否使用了正确模板？
- 输出是否覆盖必填项？
- 边界情况是否处理？
- 缺少输入时是否只问必要问题？
- 结果是否可复核？

## 性能对比

对同一任务分别执行有 Skill 和无 Skill 的版本，至少各 3-5 次，比较：

- Token 消耗。
- 用户补充和纠正次数。
- 输出质量。
- 执行时长。
- 是否稳定复现。

如果 Skill 增加了复杂度却没有减少纠正次数，应重新拆分或压缩。

## 安全测试

特别测试：

- 有副作用任务是否会自动触发。
- 提示注入能否诱导越权命令。
- Bash 白名单是否过宽。
- 是否会修改原始数据。
- 是否会泄露隐私或密钥。
- 失败时是否停止而不是继续猜测。

## 多 Skill 治理

当一个项目存在很多 Skills 时，优先治理入口预算和边界冲突。

策略：

- 保留自动触发：高频、低风险、自然语言触发、收益高的参考型 Skill。
- 改成手动触发：低频、有副作用、强授权、命令型 Skill。
- 合并：多个边界高度重叠的小 Skill。
- 拆分：一个 Skill 覆盖多个互斥领域，导致描述过宽或误触发。
- 压缩：删除 `description` 中给人看的背景故事，保留触发语义。
- 下沉：把长规范移到 `reference/`，把模板移到 `templates/`，把计算移到 `scripts/`。

不要一开始就做很多 Skill。先做少而精的高频痛点，用测试和真实反馈迭代，再扩展。

## 上线评审标准

上线前确认：

- 目录名合规。
- `SKILL.md` 全大写存在。
- `description` 能准确触发。
- 主文件不是百科全书。
- 辅助文件有加载契约。
- 权限符合最小权限。
- 有副作用任务必须手动触发。
- 有触发测试和不触发测试。
- 用户反复纠正点已写入 Skill。
