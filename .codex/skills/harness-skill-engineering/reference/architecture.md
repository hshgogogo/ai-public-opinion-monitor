# Skill 架构与 Harness 思想

## Skill 解决的问题

Skill 解决的是 Agent 的“重复入职培训”：同一套领域规则、岗位 SOP、输出格式、判断标准和工具步骤，不应在每次新会话里重新解释。

工程目标是把一次性 Prompt 变成长期资产：

- 可复用：不同任务、不同成员可重复调用。
- 可版本化：随项目、团队或行业规则同步演进。
- 可按需加载：只在相关任务出现时进入上下文。
- 可验证：能测试触发准确率、输出质量和安全边界。
- 可约束：知识不仅指导回答，也约束允许采取的行动。

## CLAUDE.md 与 Skills 的边界

| 承载物 | 加载方式 | 适合内容 | 不适合内容 |
|---|---|---|---|
| `CLAUDE.md` | 常驻上下文 | 所有任务都必须遵守的全局规则、项目背景、安全红线、通用风格 | 特定岗位 SOP、长规范、低频模板、行业公式 |
| Skill | 按需加载 | 特定领域知识、专业工作流、审查标准、报告模板、行业流程 | 每个任务都必须遵守的全局规则 |

判断法：如果这条知识“每天、每个任务都用”，放 `CLAUDE.md`；如果它“只在某类任务中用”，做成 Skill。

## Skill 是文件夹，不是一段 Prompt

一个 Skill 是一个带指令的目录，用来教 Agent 处理特定任务或工作流。这里有两个关键含义：

- “文件夹”：它可以容纳主指令、参考文档、模板、示例、脚本和验证工具。
- “教”：它不是临时命令，而是让 Agent 内化一个领域的工作范式。

标准职责拆分：

| 部分 | 职责 | 是否必需 |
|---|---|---|
| `SKILL.md` | 入口、身份、核心流程、路由表、资源加载契约 | 必需 |
| `reference/` | 长规范、领域知识、码表、标准、判断依据 | 可选 |
| `templates/` | 报告、文档、表格、消息等稳定输出格式 | 可选 |
| `examples/` | 长示例、正反案例、样例输入输出 | 可选 |
| `scripts/` | 确定性计算、扫描、转换、校验 | 可选 |

## 渐进式披露

Skill 的知识加载分三层：

1. 入口层：只扫描所有 Skill 的 `description`，判断是否相关。
2. 主文件层：命中后加载目标 `SKILL.md`，形成执行计划。
3. 附录层：执行中按契约读取具体 `reference/`、`templates/` 或 `scripts/`。

这使大部分知识留在磁盘上，只在需要时进入上下文。

## `SKILL.md` 应是路由器

不要把 `SKILL.md` 写成百科全书。主文件只应回答：

- 这个 Skill 何时使用，何时不用？
- 它按什么步骤工作？
- 用户意图如何路由到不同资源？
- 需要读取哪个文件、执行哪个脚本、使用哪个模板？
- 输出必须满足哪些约束？

把长规范、长示例、公式、模板和脚本迁出主文件。

## Harness 思想

Harness 的核心不是“更多提示词”，而是给 Agent 建一套可运行的工程支架：

- 知识层：Skill 承载领域 SOP 和专业判断。
- 行动边界：`allowed-tools` 限制可采取的动作。
- 执行层：Agent 或子智能体根据知识和工具边界完成任务。
- 验证闭环：用测试集、检查清单和迭代记录持续修正。

一句话：用工程结构驯服临场智能，让能力稳定复现。

## 设计模式

| 模式 | 适用场景 | 做法 |
|---|---|---|
| 知识分层 | 长规范、行业标准、判断准则 | 高频摘要留主文件，深知识进 `reference/` |
| 模板驱动 | 周报、合同、病历摘要、审查报告、质检表 | 固定格式进 `templates/` |
| 脚本增强 | 财务计算、数据转换、批量校验、路由扫描 | 确定性逻辑进 `scripts/` |
| 工具隔离 | 涉及写入、提交、发布、通知、外部系统 | 精确配置 `allowed-tools` |
| 复合工作流 | 多阶段、多资源、多角色协作任务 | 主文件做路由和交接契约 |
