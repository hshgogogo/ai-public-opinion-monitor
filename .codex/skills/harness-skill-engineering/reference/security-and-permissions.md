# 安全与权限设计

## 原则

`allowed-tools` 不只是权限清单，而是“知识约束行动”的机制。Skill 知道某个领域怎么做，也必须知道自己不能做什么。

核心原则：最小权限。只授予完成职责所恰好需要的工具，不多也不少。

## 权限分级

| Skill 类型 | 典型任务 | 建议权限 |
|---|---|---|
| 只读审查 | 合同审查、代码审查、病历质控、财报检查 | `Read`、`Grep`、`Glob` |
| 文档生成 | 报告、说明书、SOP、培训材料 | `Read`、`Grep`、`Glob`、必要时 `Write` |
| 文件改造 | 格式迁移、模板套用、批量修订 | `Read`、`Edit`，必要时 `Write` |
| 分析计算 | 财务比率、质检统计、库存指标 | `Read`、受限脚本命令 |
| 外部副作用 | 发布、提交、发送、删除、同步外部系统 | 手动触发，严格命令白名单 |

## Bash 白名单

Bash 权限应使用前缀白名单：

```yaml
allowed-tools:
  - Bash(git status:*)
  - Bash(git diff --stat:*)
  - Bash(python3 scripts/validate_report.py:*)
```

不要使用：

```yaml
allowed-tools:
  - Bash(*)
```

`Bash(*)` 允许任意 shell 命令，遇到提示注入、误判或幻觉时，可能造成删除文件、泄露密钥、修改历史、误发通知、误部署等真实损害。

## 副作用任务的设计

涉及真实世界状态改变时，必须满足：

- `disable-model-invocation: true`
- 用户显式调用
- 明确参数提示
- 执行前展示目标、范围、风险和当前上下文
- 命令白名单固定到团队脚本或具体子命令
- 失败路径可解释，不自动重试高风险动作

示例：

```yaml
name: deploy
description: >
  按团队批准流程发布系统。仅当用户显式调用 /deploy 并指定环境时使用。
argument-hint: "[environment] [version]"
disable-model-invocation: true
allowed-tools:
  - Read
  - Bash(git status:*)
  - Bash(git rev-parse:*)
  - Bash(./scripts/deploy-safe.sh:*)
```

部署、付款、通知、删除、外部写入等动作优先封装到团队脚本，由脚本内部做环境白名单和二次校验。

## 动态命令注入

若宿主支持 `!'command'` 这类动态上下文注入，应只用于收集上下文，如当前分支、diff 摘要、最近提交。

不建议在动态注入阶段执行测试、构建、发布、数据迁移等动作。动态注入不是执行流程本身，越早越应只读、可控、低成本。

风险点：

- 用户输入拼接到 shell 命令导致注入。
- 命令组合符如 `&&` 扩大执行范围。
- 动态阶段隐式产生副作用。
- 命令失败时上下文不完整但 Agent 继续推理。

## 行业通用安全例子

| 行业场景 | 风险 | 设计 |
|---|---|---|
| 医疗病历摘要 | 泄露隐私、误改原始记录 | 默认只读；输出脱敏；禁止外发 |
| 金融报告 | 公式错误、误写正式报表 | 计算脚本化；写入新文件；保留来源 |
| 法务合同审查 | 错改合同文本 | 只读审查或建议模式；修改需用户确认 |
| 运营群发通知 | 误发真实用户 | 手动触发；收件人预览；命令白名单 |
| 制造质检 | 错误判废或放行 | 输出建议，不直接写入生产系统 |

## 权限检查清单

- 是否真的需要写文件？
- 是否真的需要编辑已有文件？
- 是否真的需要 Bash？
- Bash 是否限定到具体命令前缀？
- 有副作用任务是否禁止自动触发？
- 是否明确“不允许做什么”？
- 是否存在用户输入拼接 shell 的路径？
- 是否能在执行前向用户展示影响范围？
