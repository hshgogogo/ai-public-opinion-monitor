# 最终项目验收

本文档定义“所有开发任务完成之后”的项目级验收流程。任务清单全部完成，只代表进入候选交付状态，不代表项目已经真正可用。

## 触发条件

当以下任一条件满足时，必须执行最终验收：

- PRD/OpenSpec/tasks 中所有任务已完成。
- 前端、后端、数据链路、异步任务、文档都已开发完毕。
- Codex 准备宣布“项目完成”“MVP 完成”“可以交付”。
- 用户明确要求最终验收、全流程测试、真实使用测试、Computer Use 测试或日志监控验收。

## 验收目标

像真实用户一样使用程序或网站，而不是只跑单元测试。

验收必须证明：

- 用户能从入口进入核心功能。
- 核心流程可以完整走通。
- 成功、失败、空状态、加载状态都清晰。
- 前端没有明显 console error。
- 网络请求、后端接口、worker、数据库或持久化链路没有隐藏错误。
- 发现问题后能回到 agent loop 修复，而不是写在总结里略过。

## 验收前准备

1. 确认当前分支不是 `main` 或 `master`，除非只做只读验收。
2. 运行常规验证：

```bash
npm test
git diff --check
git status --short
```

如果项目使用 OpenSpec：

```bash
openspec validate <CHANGE_ID> --strict
```

3. 启动应用或测试环境，按项目实际命令执行，例如：

```bash
npm run dev
docker compose up
make dev
```

4. 打开日志观察通道，按项目实际情况选择：

```bash
tail -f logs/*.log
docker compose logs -f
journalctl -f -u <service>
npm run logs
```

如果项目没有日志入口，先建立最小可用日志观察方式，至少能看到服务端请求、错误、worker 失败和关键业务事件。

## Computer Use / 浏览器真实使用

优先使用能真实操作界面的工具：

- 桌面应用：使用 Computer Use。
- 本地 Web 应用：使用 Codex Browser 或可用浏览器工具；如果用户明确要求 Computer Use，也可以用 Computer Use 操作浏览器窗口。
- 移动端或原生应用：使用可用设备、模拟器、截图和日志工具。

真实使用路径至少覆盖：

- 打开应用。
- 完成登录或进入可用状态；如果需要真实凭据，暂停请求人工确认。
- 执行核心创建/编辑/提交/查询/导出/删除等流程。
- 刷新页面或重启应用后检查状态是否保持。
- 故意触发一个失败路径，确认错误提示可理解、可操作。
- 检查关键日志和持久化结果。

## 日志监控要求

验收期间同步观察：

- 前端 console error。
- 网络请求失败、状态码异常、响应结构异常。
- 后端错误日志。
- worker、队列、定时任务错误。
- 数据库或持久化写入失败。
- 外部服务失败、重试、限流、鉴权错误。

所有日志问题必须分类：

- P0：安全、数据破坏、凭据泄露、生产风险。立即停止。
- P1：核心流程不可用。必须修。
- P2：用户体验、可维护性或可观测性明显不足。验收前修。
- P3：不阻塞交付，但必须记录。

## 问题回流

最终验收发现 P0/P1/P2 时，不允许宣布项目完成。必须回到 agent loop：

```text
验收问题
-> 缺陷切片
-> done rubric
-> 失败测试或复现脚本
-> 修复
-> 验证
-> evidence report
-> 重新最终验收
```

每个回流问题都应有：

- 复现步骤。
- 期望行为。
- 实际行为。
- 日志或截图证据。
- 修复验证命令。
- 是否需要新增 memory lesson。

## 完成判定

只有满足以下条件，才能宣布项目完成：

- 所有任务完成。
- 常规验证通过。
- 真实使用全流程通过。
- 日志监控未发现未解决的 P0/P1/P2。
- 所有验收发现的问题已经回流修复并重新验收。
- 最终验收报告已写入 PR、交付说明或项目文档。
